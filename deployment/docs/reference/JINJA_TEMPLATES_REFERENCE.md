## Application Configuration

### env.j2

**Purpose:** Auto-generated environment file with Flask configuration, AWS integration, and service names

**Location:** `{{ app_dir }}/.env` (auto-generated, don't edit)

**Key Variables:**
- `FLASK_ENV`: Flask environment mode (production/development)
- `SECRET_KEY`: Session signing key (from vault.yml)
- `SECRET_NAME`: AWS Secrets Manager secret name
- `AWS_REGION`: AWS region for services
- `S3_BUCKET_NAME`: S3 bucket for data storage
- `CLOUDWATCH_NAMESPACE`: CloudWatch metrics namespace
- `EBAY_VERIFICATION_TOKEN`: eBay webhook verification (persisted)

**Security Features:**
- ✓ No hardcoded secrets (uses AWS Secrets Manager)
- ✓ Three-tier fallback (Secrets Manager → env vars → defaults)
- ✓ Safe to check into source control

**When to edit:**
- Edit `group_vars/all.yml` and `group_vars/vault.yml` instead
- Re-run `setup.yml` to regenerate

**Troubleshooting:**
```bash
# View current .env file
cat {{ app_dir }}/.env

# Check if app can access Secrets Manager
aws secretsmanager get-secret-value --secret-id {{ app_name }}/secrets

# View generated values
grep SECRET_NAME {{ app_dir }}/.env
```

## Web Server & Reverse Proxy

### nginx.conf.j2

**Purpose:** Reverse proxy, TLS termination, rate limiting, and WAF-style protection

**Location:** `/etc/nginx/sites-available/{{ app_name }}` → linked to `sites-enabled/`

**Key Features:**
- ✓ Automatic SSL detection (uses existing Let's Encrypt cert if available)
- ✓ HTTP→HTTPS redirect (when SSL enabled)
- ✓ Rate limiting zones (login: 20/min, API: 200/min, general: 300/min)
- ✓ Security headers (HSTS, CSP, X-Frame-Options, etc.)
- ✓ Path-based WAF (blocks common attacks)
- ✓ Static file caching (30-day expires)
- ✓ Large upload support (5GB)

**Rate Limiting Zones:**
```
- login_limit: 20 requests/minute (brute force protection)
- api_limit: 200 requests/minute (API rate limiting)
- general_limit: 300 requests/minute (page views)
```

**Security Headers:**
```
- X-Frame-Options: SAMEORIGIN (clickjacking protection)
- Content-Security-Policy: Strict CSP with S3/eBay allowlist
- HSTS: 1-year strict transport security
- X-XSS-Protection: XSS protection enabled
- Referrer-Policy: Strict origin when cross-origin
```

**Blocked Paths (WAF):**
- Admin paths: `/admin`, `/phpmyadmin`, `/wp-admin`
- Sensitive files: `.env`, `.git`, `.aws`, etc.
- Path traversal attempts: encoded slashes, double-encoding
- Scanner paths: `@fs`, `__vite`, `node_modules`, etc.

**When to edit:**
- For new application routes: Add location block before the final deny
- For stricter CSP: Modify the `add_header Content-Security-Policy` line
- For different rate limits: Edit the `limit_req_zone` lines at top

**Testing:**
```bash
# Syntax check
sudo nginx -t

# View active config
sudo nginx -T | grep server_name

# Monitor requests in real-time
tail -f /var/log/nginx/{{ app_name }}_access.log

# Check rate limiting
grep "limit_req" /var/log/nginx/{{ app_name }}_error.log
```

## Process Management

### systemd-with-validation.service.j2

**Purpose:** Service unit with pre-start CSV schema validation

**Location:** `/etc/systemd/system/{{ app_name }}.service`

**Best For:**
- Applications that require data validation before startup
- Apps using CSV data files
- Strict reliability requirements

**Pre-start Validation:**
```bash
# These run before ExecStart and must succeed (exit 0)
ExecStartPre: echo "Validating CSV schema..."
ExecStartPre: python util-validate-csv-schema.py --quiet
ExecStartPre: echo "✓ CSV schema validation passed"
```

**Gunicorn Configuration:**
- Workers: 2 (handles 2 concurrent long-running requests)
- Threads: 4 per worker (8 concurrent total)
- Timeout: 120 seconds
- Worker class: gthread (threading for I/O-bound tasks)

**Usage:**
```bash
systemctl start {{ app_name }}
systemctl stop {{ app_name }}
systemctl restart {{ app_name }}
systemctl status {{ app_name }}
journalctl -u {{ app_name }} -f  # Real-time logs
```

**When to use:**
- Use this if you have CSV data files that must be valid
- Use this if you want strict validation before app starts

### app.service.j2

**Purpose:** Service unit with comprehensive security hardening

**Location:** `/etc/systemd/system/{{ app_name }}.service`

**Best For:**
- Maximum security posture
- Production deployments
- Apps handling sensitive data

**Security Hardening (Multiple Layers):**

1. **Privilege Restrictions**
   - `NoNewPrivileges=true`: Can't use setuid binaries
   - `RestrictSUIDSGID=true`: Can't execute SUID/SGID

2. **Filesystem Isolation**
   - `ProtectHome=read-only`: /home is read-only
   - `ProtectSystem=strict`: /, /usr, /boot, /etc read-only
   - `ReadWritePaths`: Only specific dirs writable (logs, instance)

3. **System Call Filtering**
   - `SystemCallFilter=@system-service`: Whitelist approach
   - `SystemCallFilter=~@privileged @resources @mount`: Remove dangerous syscalls
   - Kills process if blocked syscall attempted

4. **Device & Network**
   - `PrivateDevices=true`: No raw device access
   - `RestrictAddressFamilies=AF_UNIX AF_INET AF_INET6`: Only normal sockets

5. **Process Isolation**
   - `ProtectProc=invisible`: Other processes invisible
   - `ProtectHostname=true`: Can't change hostname
   - `RestrictNamespaces=true`: Can't unshare namespaces

6. **Capabilities**
   - `CapabilityBoundingSet=`: Remove ALL Linux capabilities
   - Process has minimum privileges needed

**Performance Impact:**
- <1% overhead
- Security gain is significant
- Recommended for production

**Usage (Same as systemd-with-validation):**
```bash
systemctl start {{ app_name }}
systemctl status {{ app_name }}
journalctl -u {{ app_name }} -f
```

**When to use:**
- Use this for maximum security (recommended)
- Use this in production deployments
- Use this unless you need pre-start validation

## Monitoring & Logging

### cloudwatch-config.json.j2

**Purpose:** CloudWatch agent configuration for logs and metrics

**Location:** `/opt/aws/amazon-cloudwatch-agent/etc/config.json`

**Logs Collected:**
- Application logs: `/var/log/{{ app_name }}/app.log`
- Error logs: `/var/log/{{ app_name }}/error.log`
- Nginx access: `/var/log/nginx/{{ app_name }}_access.log`
- Nginx errors: `/var/log/nginx/{{ app_name }}_error.log`

**Metrics Collected:**
- CPU usage (idle, iowait percentages)
- Memory usage (percent used)
- Disk usage (percent used)

**Log Groups:**
```
/{{ app_name }}/application   # App logs
/{{ app_name }}/error         # App errors
/{{ app_name }}/nginx-access  # HTTP requests
/{{ app_name }}/nginx-error   # Server errors
```

**Retention Policy:**
- Default: 30 days (configurable via `log_retention_days`)
- Automatic deletion of old logs

**CloudWatch Insights Queries:**
```bash
# Find errors in last hour
fields @timestamp, @message | filter @message like /ERROR/ | stats count() by bin(5m)

# Find slow requests (>1 second)
fields @duration | filter @duration > 1000 | stats avg(@duration), max(@duration) by bin(1m)

# Find 4xx/5xx errors
fields @timestamp, httpStatus | filter httpStatus >= 400 | stats count() by httpStatus
```

**Viewing Logs:**
```bash
# List log groups
aws logs describe-log-groups --region {{ aws_region }}

# Get log streams
aws logs describe-log-streams --log-group-name /{{ app_name }}/application

# Query logs
aws logs filter-log-events --log-group-name /{{ app_name }}/application \
  --filter-pattern "ERROR" --region {{ aws_region }}
```

**Cost:**
- Log ingestion: $0.50/GB
- Storage: $0.03/GB/month
- Metrics: $0.30/metric/month
- Queries: $0.005/query

### logrotate.conf.j2

**Purpose:** Rotate application logs to prevent disk overflow

**Location:** `/etc/logrotate.d/{{ app_name }}`

**Rotation Policy:**
- Frequency: Daily
- Trigger: Daily OR 100MB (whichever first)
- Retention: 20 rotated files
- Compression: gzip (saves ~90% space)
- Format: dateext (app.log-20240223, app.log-20240222.gz)

**File Lifecycle:**
```
Day 1: app.log (0-100MB written)
       logrotate runs
Day 2: app.log (fresh, 0 bytes)
       app.log-20240223 (yesterday's logs)
       logrotate compresses yesterday's file
Day 3: app.log (current)
       app.log-20240223 (compressed to .gz)
       app.log-20240222.gz
```

**Disk Space Example:**
```
Without rotation: 
  - app.log grows unbounded
  - Could exceed available disk space
  - App crashes when disk full

With rotation (20 files @ ~10MB each):
  - Total disk: ~200MB for logs
  - Safe, predictable disk usage
  - Old logs automatically deleted
```

**Monitoring:**
```bash
# View rotation logs
tail -f /var/log/logrotate.status

# Check file sizes
du -sh /var/log/{{ app_name }}/*.log

# See compression ratio
ls -lh /var/log/{{ app_name }}/ | head -20
```

### log-monitor.sh.j2

**Purpose:** Health monitoring script that checks app health and sends alerts

**Location:** `/opt/{{ app_name }}/bin/log-monitor.sh`

**What it Checks:**
1. Error count in last 5 minutes
   - If > threshold: Sends SNS alert
2. Service status via systemctl
   - If down: Restarts service and alerts
3. Cleans old monitor logs (>30 days)

**Configuration:**
- `ERROR_THRESHOLD`: 10 (errors before alerting)
- `sns_topic_arn`: Optional SNS topic for alerts

**Manual Usage:**
```bash
# Run health check manually
/opt/{{ app_name }}/bin/log-monitor.sh

# Check monitoring logs
tail -f /var/log/{{ app_name }}/monitor.log
```

**Automated Monitoring (Cron):**
```bash
# Add to crontab to check every 5 minutes
*/5 * * * * /opt/{{ app_name }}/bin/log-monitor.sh

# View cron logs
grep {{ app_name }} /var/log/syslog
```

**Alerts Sent:**
- Subject: "{{ app_display_name }}: High Error Rate" (many errors detected)
- Subject: "{{ app_display_name }}: Service Restarted" (service was down)

**Troubleshooting:**
```bash
# Make script executable
chmod +x /opt/{{ app_name }}/bin/log-monitor.sh

# Test SNS access
aws sns publish --topic-arn "arn:aws:sns:..." --subject "Test" --message "Test" \
  --region {{ aws_region }}
```

## Security Hardening

### jail.local.j2

**Purpose:** Fail2ban intrusion detection and prevention

**Location:** `/etc/fail2ban/jail.local`

**How It Works:**
1. Monitors log files for attack patterns
2. Counts failed attempts per IP
3. When threshold reached: Adds IP to firewall blacklist (iptables)
4. IP remains banned for configured duration
5. Automatically unbans after time expires

**Jails Enabled:**
- `[sshd]`: SSH brute force (3 attempts, 24-hour ban)
- `[nginx-http-auth]`: HTTP basic auth failures
- `[nginx-badbots]`: Known malicious bots (1 attempt, 24 hours)
- `[nginx-botsearch]`: Automated scanning
- `[nginx-limit-req]`: Nginx rate limit hits (429 responses)
- `[nginx-404]`: Excessive 404 errors
- `[nginx-403]`: Excessive 403 forbidden
- `[recidive]`: Repeat offenders (7-day ban)

**Monitoring:**
```bash
# Check jail status
fail2ban-client status                    # All jails
fail2ban-client status sshd               # Specific jail

# See banned IPs
iptables -L -n | grep -A 10 fail2ban

# View ban history
tail -f /var/log/fail2ban.log

# Check current bans
fail2ban-client status sshd | grep "Banned IP list:"
```

**Manual Control:**
```bash
# Unban an IP
fail2ban-client set sshd unbanip 192.168.1.100

# Ban an IP manually
fail2ban-client set sshd banip 192.168.1.100

# Reload rules
fail2ban-client reload
```

**Adjust Thresholds:**
Edit `group_vars/all.yml`:
```yaml
fail2ban_maxretry: 3           # Attempts before ban
fail2ban_bantime: 86400        # Seconds (24 hours)
fail2ban_findtime: 1200        # Time window (20 minutes)
```

### 20auto-upgrades.j2 & 50unattended-upgrades.j2

**Purpose:** Automatic security updates for system packages

**Locations:**
- `/etc/apt/apt.conf.d/20auto-upgrades` (enables daily updates)
- `/etc/apt/apt.conf.d/50unattended-upgrades` (controls what's updated)

**How It Works:**
1. `20auto-upgrades.j2` enables the APT update process
2. Daily cron job runs `apt update` and `apt upgrade`
3. `50unattended-upgrades.j2` specifies which packages to update
4. Security patches: Automatically installed
5. Infrastructure patches: Skipped (prevent breaking changes)
6. Optional: Auto-reboot if kernel updated

**Configuration:**
- `Automatic-Reboot`: Set to "false" (requires manual reboot confirmation)
- `Automatic-Reboot-Time`: "03:00" (middle of the night if enabled)
- `Remove-Unused-Kernel-Packages`: "true" (cleanup old kernels)

**Monitoring:**
```bash
# View update history
cat /var/log/apt/history.log

# See what's queued for update
apt list --upgradable

# Check unattended-upgrades logs
tail -f /var/log/unattended-upgrades/unattended-upgrades.log

# View applied updates
grep "^Unpacking" /var/log/apt/history.log
```

**Manual Updates:**
```bash
# Check for updates without installing
apt update && apt list --upgradable

# Install all updates manually
apt upgrade

# Install security updates only
apt install -o APT::AutoRemove::SuggestsImportant=false -o APT::AutoRemove::RecommendsImportant=false $(apt list --upgradable 2>/dev/null | cut -d'/' -f1)
```

### rsyslog-rotation.j2

**Purpose:** Rotate system logs to prevent disk overflow

**Location:** `/etc/logrotate.d/rsyslog`

**Logs Rotated:**
- `/var/log/syslog`: General system messages
- `/var/log/auth.log`: SSH, sudo, login attempts
- `/var/log/kernel.log`: Kernel messages
- `/var/log/mail.*`: Mail server logs
- `/var/log/daemon.log`: System daemons

**Rotation Policy:**
- Frequency: Daily
- Retention: 7 rotated files (1 week)
- Compression: gzip
- Filename format: `syslog.1`, `syslog.2.gz`, etc.

**Why Important:**
- Auth logs: Track who's accessing the system
- System logs: Diagnose kernel issues, driver problems
- Mail logs: Troubleshoot email delivery

**Viewing Logs:**
```bash
# Current logs
tail -f /var/log/auth.log          # SSH, login attempts
tail -f /var/log/syslog            # System messages
tail -f /var/log/kernel.log        # Kernel messages

# Archived logs (compressed)
zcat /var/log/auth.log.2.gz | tail -20
```

## Process Manager (Alternative)

### supervisor.conf.j2

**Purpose:** Alternative to systemd for process management

**Location:** `/etc/supervisor/conf.d/{{ app_name }}.conf`

**When to Use:**
- When systemd is unavailable or problematic
- For containers using supervisor as init
- Legacy systems preferring supervisor

**Features:**
- Auto-restart on crash
- Process monitoring
- Easy start/stop/restart via supervisorctl
- Logging to single file

**Usage:**
```bash
# Interactive CLI
supervisorctl

# Commands
supervisorctl status {{ app_name }}
supervisorctl start {{ app_name }}
supervisorctl stop {{ app_name }}
supervisorctl restart {{ app_name }}
supervisorctl tail {{ app_name }}
supervisorctl tail {{ app_name }} -f    # Follow logs

# Reload config
supervisorctl reread
supervisorctl update
```

**Note:** Not recommended if systemd is available (systemd is more powerful and integrates better with modern Linux)

## Template Variables Reference

### Universal Variables (All Templates)

```yaml
# Application identity
app_name: "{{ app_name }}"                    # Technical name (lowercase)
app_display_name: "{{ app_display_name }}"    # Display name
service_name: "{{ service_name }}"            # Systemd/supervisor service name

# Directories (all on EBS volume)
app_dir: "{{ app_dir }}"                      # Application root
app_storage_mount_path: "{{ app_storage_mount_path }}"  # /{app_name}
venv_dir: "{{ venv_dir }}"                    # Python virtual env
log_dir: "{{ log_dir }}"                      # Application logs

# User/Group
app_user: "{{ app_user }}"                    # App runtime user
admin_user: "{{ admin_user }}"                # Admin/deployment user

# AWS Configuration
aws_region: "{{ aws_region }}"                # AWS region
s3_bucket_name: "{{ s3_bucket_name }}"        # S3 bucket name
secret_name: "{{ secret_name }}"              # Secrets Manager secret

# Server Configuration
server_name: "{{ server_name }}"              # Domain or "_" for IP-only
```

### Template-Specific Variables

**nginx.conf.j2:**
- `server_name`: Domain for SSL cert path
- `service_name`: Log filename prefix

**systemd & app.service.j2:**
- `app_user`: Run as this user
- `app_dir`: Working directory
- `venv_dir`: Python binaries
- `log_dir`: Log output
- `gunicorn_workers`: Number of workers
- `gunicorn_threads`: Threads per worker
- `gunicorn_timeout`: Request timeout (seconds)
- `gunicorn_bind`: Bind address

**CloudWatch:**
- `log_retention_days`: How long to keep logs (default: 30)
- `sns_topic_arn`: Optional SNS alerts

## Best Practices

### Do's ✅

1. **Never manually edit generated templates** - They'll be overwritten on redeploy
2. **Edit group_vars/all.yml instead** - Then re-run playbooks to regenerate
3. **Keep backups of .env and vault.yml** - Before major redeployments
4. **Review template changes** - Before deploying to production
5. **Test in staging first** - Try new configurations on test instance
6. **Monitor logs** - Use CloudWatch or `tail -f` for real-time visibility

### Don'ts ❌

1. **Don't manually edit /etc/nginx/sites-available/** - Use the template
2. **Don't modify /etc/systemd/system/ files** - Regenerate from template
3. **Don't put secrets in .env** - Use AWS Secrets Manager
4. **Don't check vault.yml into git** - Keep it encrypted and local
5. **Don't disable security hardening** - It has minimal performance impact
6. **Don't ignore log rotation** - Disk space issues can cause outages

## Troubleshooting Templates

### Common Issues

**Problem: "permission denied" on .env file**
```bash
# Fix: Ensure correct permissions
chmod 640 {{ app_dir }}/.env
chown {{ app_user }}:{{ app_user }} {{ app_dir }}/.env
```

**Problem: Nginx config syntax error**
```bash
# Test nginx config
sudo nginx -t

# If error, check for template issues
sudo nginx -T | grep -A 5 -B 5 "error"
```

**Problem: App won't start (systemd failure)**
```bash
# Check systemd logs
journalctl -u {{ app_name }} -n 50

# Check if validation failed
journalctl -u {{ app_name }} | grep "validation"

# Try starting manually to see errors
sudo systemctl start {{ app_name }} --no-block
journalctl -u {{ app_name }} -f
```

**Problem: CloudWatch logs not appearing**
```bash
# Check agent status
sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \
  -a status -m ec2 -config-location /opt/aws/amazon-cloudwatch-agent/etc/config.json

# Restart agent
sudo systemctl restart amazon-cloudwatch-agent

# Check agent logs
tail -f /opt/aws/amazon-cloudwatch-agent/logs/amazon-cloudwatch-agent.log
```

## Template Customization Guide

### Adding Custom Application Routes (nginx.conf.j2)

```nginx
# Add BEFORE the final "deny all" location block
location /my-custom-endpoint {
    limit_req zone=general_limit burst=50 nodelay;
    limit_req_status 429;

    proxy_pass http://127.0.0.1:8000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;

    # Prevent buffer overflow attacks
    proxy_buffer_size 4k;
    proxy_buffers 8 4k;
    proxy_busy_buffers_size 8k;

    # Timeout settings
    proxy_connect_timeout 300s;
    proxy_send_timeout 300s;
    proxy_read_timeout 300s;
}
```

### Adjusting Gunicorn Workers (systemd templates)

Edit `group_vars/all.yml`:
```yaml
# More workers = more concurrent requests (needs RAM)
# Rule of thumb: 2-4 workers per CPU core
gunicorn_workers: 4
gunicorn_timeout: 120
```

Then regenerate: `ansible-playbook setup.yml`

### Changing Log Retention (CloudWatch)

Edit `group_vars/all.yml`:
```yaml
log_retention_days: 7  # Keep only 1 week
# or
log_retention_days: 90  # Keep 3 months
```

Then regenerate: `ansible-playbook setup-monitoring.yml`
